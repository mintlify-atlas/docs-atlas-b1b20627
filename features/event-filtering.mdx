This mapping is defined in `/workspace/source/web-app/src/lib/constants.ts:276-309` and ensures consistent display formatting across the application.

## How event filtering works

Event filtering performs an exact match query against the `event` column in the projects table:

```typescript
if (selectedEvent) {
  query = query.eq('event', selectedEvent);
  countQuery = countQuery.eq('event', selectedEvent);
}
```

Each project is associated with exactly one event where it was submitted.

## URL parameters

Event selections are reflected in the URL using the `event` parameter:

<CodeGroup>
```text Prague 2025
https://ethglobalexplorer.com/?event=ETHGlobal%20Prague
```

```text Bangkok 2024
https://ethglobalexplorer.com/?event=ETHGlobal%20Bangkok
```

```text Trifecta - Agents
https://ethglobalexplorer.com/?event=ETHGlobal%20Trifecta%20-%20Agents
```

```text Combined with search
https://ethglobalexplorer.com/?q=wallet&event=ETHGlobal%20Singapore
```
</CodeGroup>

<Info>
The event parameter uses the full internal event name (e.g., `ETHGlobal Prague`), not the shortened display name (`Prague '25`).
</Info>

## Event data structure

Events are stored as two separate arrays in the constants file:

1. **allEvents** (`constants.ts:1`): Array of all event names in chronological order
2. **eventDisplayNames** (`constants.ts:276`): Mapping object for shortened display names

Events without a display name mapping fall back to showing their full stored name.

## Combining with other filters

Event filtering works alongside:
- **Search**: Find specific projects within an event
- **Prize filtering**: See which projects won prizes at a specific event
- **Tag filtering**: Explore themed projects (AI, NFT, DAO) from a particular hackathon

<Note>
Some older events may have fewer projects as the database was populated from more recent data scraping efforts.
</Note>
